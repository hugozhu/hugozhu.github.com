从大厂螺丝钉到 OPC 公司老板：你只缺了三个 FDE 项目

Three FDE deliveries to go from cog to one-person-team founder

 Posted on June 30, 2026  |  2 min  |  407 words

上个月一个前同事找我吃饭。他在某大厂做了八年后端，刚被裁。手里有两个选择：去创业公司做 Agent 工程师，或者自己干。

他问我：「自己干的话，我该做什么？」

我说：你做过钉钉 FDE 那种到客户现场交付 Agent 的项目吗？

他说没有，只在公司内部做过。

我说：那你差的就是三个项目。做完三个，你就可以一个人开公司了。

他愣了一下：「三个就够了？」

够了。但不是随便三个。

为什么是三个，不是三十个

三个项目不是随口一说。它对应 FDE 能力曲线上的三个拐点，每个拐点解锁一层新的交付效率：

项目 1：破冰（Learn）
  交付成本：1x（基线）
  核心收获：端到端能力——业务发现 → 工作流抽象 → Agent 设计 → 客户验证

项目 2：复制（Leverage）
  交付成本：≤ 0.5x（减半）
  核心收获：模式识别——复用项目 1 的框架，换行业或换场景

项目 3：产品化（Scale）
  交付成本：≤ 0.25x（再减半）
  核心收获：标准化——把前两个项目的共性抽象成可售卖的 Agent 模板

三个项目跑完，你的交付效率是起点的 4 倍，利润率从 30% 涨到 80%。 你不再是「按项目卖时间」，而是「按模式卖能力」。 这就是 OPC（One Person Company）的经济学基础。

为什么不是三十个？因为到第四个项目时，你已经进入稳态——边际收益递减，剩下的项目只是扩大客户群，不再解锁新能力。

大厂螺丝钉的困境

先说清楚，螺丝钉不是贬义词。在大厂做螺丝钉意味着你的能力被平台封装了：

维度大厂螺丝钉OPC 老板
能力边界只做链路中的一环端到端全链路
价值归属归平台（你做的功能写着公司名字）归自己（客户认的是你这个人）
客户关系隔着 PM、售前、实施直接面对决策者
交付粒度功能模块完整方案
定价权没有（工资由 HR 定）有（按交付价值定价）
可迁移性低（平台特有技术栈）高（通用 Agent 模式）

螺丝钉不是能力不行，是 被平台限制了能力的变现路径。

我在 从做网站到做 Agent：工程师没变，交付物变了 中讨论过这个趋势：当 AI Coding 让开发成本趋近于零，工程师的价值从「支撑的平台规模有多大」翻转为「个性化交付的成本有多低」。

OPC 就是这个翻转的个人版本——你不需要支撑千万 DAU，你只需要能以极低成本交付个性化 Agent 方案。

FDE 模式为什么是桥梁

为什么 FDE（Forward Deployed Engineer）模式特别适合个人创业？因为它的四个特征天然匹配 OPC 的约束条件：

1. 轻资产。 不需要产品团队、不需要融资、不需要销售团队。一个人 + AI Coding + 平台基础设施（钉钉、飞书、企微）= 完整交付能力。我在 钉钉 FDE 的一天 中记录过一个 FDE 从业务调研到 Agent 上线的全流程——整个过程一个人一天就能完成。

2. 快验证。 一个项目 3-5 天交付，客户当场验证价值。不需要 6 个月的开发周期，不需要写 PRD 等审批。做完就用，用了就知道有没有效。

3. 模式沉淀。 每个项目都在积累可复用的 Agent 模式。第 3 个项目交付时，70% 的代码来自前两个项目的模式库。这和 Token 交付时代 的逻辑一致——当交付的颗粒度从「软件产品」缩小到「Agent 模式」，每次交付的边际成本就趋近于零。

4. 客户即渠道。 一个成功的 FDE 交付，客户会主动推荐给同行。你帮他做的报价 Agent 太好用了，他的朋友也想要——这就是 OPC 的获客方式，不需要投广告、不需要 BD 团队。

交付成本减半的数学

具体算一笔账。假设基线项目的成本 = 100（人天 × 单价）：

项目 1：成本 100
  ├── 从零开始，边学边做
  ├── 实际人天：5 天
  ├── 模式沉淀：1 个 Agent 模板 + 工作流抽象方法论
  └── 利润率：~30%（还在交学费）

项目 2：成本 ≤ 50
  ├── 复用项目 1 的模式，新做差异化部分
  ├── 实际人天：2-3 天
  ├── 模式沉淀：行业 Agent 库 + 配置化交付框架
  └── 利润率：~60%（复用生效）

项目 3：成本 ≤ 25
  ├── 复用 75%，新做 25%
  ├── 实际人天：1-2 天
  ├── 模式沉淀：标准化产品 + 自助配置工具
  └── 利润率：~80%（几乎纯利润）

关键洞察：你收费可以不变——因为客户付的是价值，不是你的时间。 项目 1 你收 5 万，项目 3 你还是收 5 万，但你的实际成本从 3.5 万降到了 1 万。这就是 代码复制成本归零 在个人层面的体现——你的模式库就是零边际成本的复制资产。

当然，这些数字是基于我观察到的 FDE 交付案例的经验估算，具体比例因行业和客户复杂度而异。但递减趋势是确定的。

从螺丝钉到 OPC 的四阶段路线图

Phase 0（在职准备，1-2 个月）
  ├── 掌握 AI Coding 工具（Claude Code / Cursor / OpenCode）
  ├── 深入理解一个 AI 平台的全栈能力（审批、表格、工作流、Agent）
  ├── 做出 2-3 个 demo Agent，证明端到端交付能力
  └── 找到第一个付费客户（前同事、朋友的公司、行业社群）

Phase 1（第一个项目，1-2 周）
  ├── 业务发现（1-2 天）：跟客户一起梳理 SOP，找到 Agent 能切入的环节
  ├── Agent 设计（1 天）：定义工具 + 工作流 + 记忆范围
  ├── 编码交付（1-2 天）：AI Coding 快速实现
  ├── 客户验证（1 天）：现场跑通 + 反馈修正
  └── 复盘沉淀（1 天）：提炼模式、记录方法论

Phase 2（第二、三个项目，各 3-5 天）
  ├── 复用模式库，配置化交付
  ├── 聚焦差异化部分（行业特定逻辑、特殊数据源）
  └── 持续沉淀，扩大模式库覆盖范围

Phase 3（OPC 常态化）
  ├── 每月 2-4 个项目，月收入稳定
  ├── 模式库覆盖 80% 常见场景
  ├── 口碑获客，不需要主动销售
  └── 可选：把高频模式做成 SaaS 产品，从项目制转向订阅制

Phase 0 到 Phase 1 是最难的一步——你需要在还有工资保障的时候，挤出时间准备和找到第一个客户。我的建议是： 不要裸辞。 用业余时间做出 demo，找到愿意付费的第一个客户后再全职切换。

什么样的人适合这条路

不是所有螺丝钉都该变成 OPC 老板。这条路适合三种人：

1. 全栈能力被低估的人。 你在大厂只负责链路中的一环，但你其实能端到端做完一个项目。FDE 模式让你把被平台封印的能力释放出来。

2. 对业务有直觉的人。 你不只会写代码，你能看到客户的 SOP 哪里可以用 Agent 优化。这种业务直觉在 FDE 模式中比纯技术能力更值钱——我在 FDE 基础设施实战 中记录过，FDE 能力模型里业务抽象能力占 40% 权重，比编码能力的 20% 高一倍。

3. 厌恶办公室政治但不厌恶工作的人。 你不是不想干活，你是不想把时间花在开会、汇报、抢资源上。OPC 模式下你的时间 100% 花在交付价值上。

一个诚实的提醒

这条路也有风险，值得提前想清楚：

收入不稳定。 项目制意味着没项目就没收入。前期需要 3-6 个月的现金流储备。

获客有冷启动问题。 口碑传播需要时间，第一批客户最难找。如果你在大厂完全没有行业人脉，Phase 0 会更长。

孤独感。 OPC 意味着没有人帮你扛。技术方案自己想，客户关系自己维护，出了问题自己兜底。

模式库的保质期。 Agent 技术在快速迭代，你今天沉淀的模式可能半年后就过时了。持续学习是 OPC 的隐性成本。

这些不是劝退，是校准。知道代价才能做清醒的选择。

回到开头

那个前同事后来怎么样了？

他花了两个月准备（Phase 0），找到了一个做外贸的朋友当第一个客户。项目是做一个询盘自动分类和回复的 Agent。5 天交付，收了 3 万。

第二个月又接了两个——一个做物流的，一个做跨境电商的。他发现这三个客户的 Agent 底层逻辑惊人地相似：都是「接收信息 → 分类 → 按规则回复」。他抽象出了一个通用模板，第三个项目只花了 1 天。

现在他每月稳定做 3-4 个项目，月收入比以前在大厂高 50%，工作时间少一半。

他不是天才，不是连续创业者，就是一个做了八年后端、终于找到把能力直接卖给客户的方法的普通工程师。

从螺丝钉到 OPC 老板，中间隔的不是能力差距，而是三个 FDE 项目的经验差距。 这个差距，用 3-6 个月就能补上。

你有没有想过从大厂出来自己干？最大的顾虑是什么？欢迎留言讨论。
